## Active conventions worth preserving

### Branch naming
- `v<N>-<short-name>` for feature branches (e.g. `v06-review-effort`, `v07-setup-probe`).
- `v<N>-release-bump` for the chore release PR.

### Release process
```bash
# After feature PRs merged to main:
node scripts/bump-version.mjs 0.0.N       # bumps 3 version strings (package-lock.json is optional)
# Edit plugins/copilot/CHANGELOG.md: prepend a new 0.0.N section with Added/Changed/Fixed bullets
git add package.json plugins/copilot/.claude-plugin/plugin.json .claude-plugin/marketplace.json plugins/copilot/CHANGELOG.md
git commit -m "chore(release): bump version to 0.0.N"
git push -u origin v<N>-release-bump
gh pr create ...                          # release PR
# After CI green on release PR:
gh pr merge <n> --squash --delete-branch --admin
git checkout main && git pull
MERGE_SHA=$(git rev-parse HEAD)
git tag -a v0.0.N $MERGE_SHA -m "v0.0.N — <one-line theme>"
git push origin v0.0.N
gh release create v0.0.N --title "v0.0.N" --notes "$(<CHANGELOG excerpt>)"
```

### Test spawn pattern
- **Always** `run(process.execPath, ...)` for Node-in-tests, never bare `"node"`. The Windows shell wrapper in `tests/helpers.mjs` splits on spaces in the absolute path (`OneDrive - Microsoft` → `OneDrive`), causing `MODULE_NOT_FOUND`. See PR #14.

### Fake Copilot (`tests/fake-copilot.mjs`) capabilities
- `--version` probe (short-circuit exit 0, skipped by SPAWN_LOG).
- `--acp` mode (full ACP v1 JSON-RPC; driven by `FAKE_COPILOT_SCRIPT`).
- `-p <prompt>` CLI mode (mirrors real `copilot -p`; emits concatenated `agent_message_chunk` text).
- `FAKE_COPILOT_SPAWN_LOG` → writes one JSON line per spawn with argv + selected env.
- `script.unavailableModels: [...]` → in `-p` mode, invoked models in this list exit 1 with `"Error: model X is not available on this Copilot account."` (targets `isModelUnavailableStderr` regex).

### Security deny-list
`SHELL_METACHAR_RE` + `assertNoShellMetachars` in `plugins/copilot/scripts/lib/copilot.mjs`:
- Scoped to the `shell:true` spawn path only (Windows production with the real `.cmd` launcher).
- Under `shell:false` (Linux/macOS production, all tests), argv passes to `CreateProcess` / `execve` verbatim — no injection risk, deny-list skipped.
- The regex was hardened in v0.5 fixup to reject bare `forbidden` (common in content-policy stderr) and in v0.6 to use `[^\n]` instead of `[\s\S]` for the gap (no cross-paragraph matching).

### Fallback chain
`EFFORT_FALLBACK_CHAIN` in `plugins/copilot/scripts/copilot-companion.mjs`:
- `--effort high` / `xhigh`: `claude-opus-4.6` → `claude-sonnet-4.5` → `claude-opus-4.6-fast`
- `--effort medium`: `claude-sonnet-4.5` → `claude-opus-4.6-fast`
- `--effort low` / `minimal` / `none`: already lowest, no fallback
- `--resume-last` collapses to single-element (broker ignores per-call `--model` anyway).
- Explicit `--model X` **never** triggers fallback — user's deliberate pick.
- Applies uniformly across `/copilot:task`, `/copilot:review`, `/copilot:adversarial-review` as of v0.0.6.

### Availability detection
`isModelUnavailableStderr(text)` in `lib/copilot.mjs`:
- Requires `\bmodel\b` + a specific availability phrase on the same line within ~80 chars.
- Bare `forbidden` is **not** an indicator (content-policy false positives).
- Unit-tested in `tests/is-model-unavailable-stderr.test.mjs` (positive + negative cases).

## v1.1 deferred items (next candidates)

From the design doc's "Deferred to v1.1" list. All three are substantial features needing their own design passes.

### 1. Concurrent broker reuse across Claude sessions on the same workspace

**Current state:** each Claude session spawns its own `copilot --acp` broker, discovered via `broker.json` under `<CLAUDE_PLUGIN_DATA>/state/<workspace>/`. Multi-session coordination on the same workspace is untested — a second Claude session would either race on broker creation or try to talk to a broker it doesn't own.

**Design sketch:** handshake + shared-secret lock in `broker-lifecycle.mjs::ensureBrokerSession`. The broker already has a 256-bit `BROKER_SECRET_ENV` secret that's required in the `initialize` handshake, so the auth primitive exists. What's missing is cross-session *coordination*: who creates the broker when two sessions start simultaneously, who owns the teardown when the first session ends, and how to detect a dead-broker `broker.json` that another session claimed.

**Key files:**
- `plugins/copilot/scripts/lib/broker-lifecycle.mjs` — `ensureBrokerSession`, `spawnBrokerProcess`, `teardownBrokerSession`.
- `plugins/copilot/scripts/acp-broker.mjs` — the detached broker process.
- `tests/broker-lifecycle.test.mjs` — extend with two-session scenarios.

**Suggested first step:** add a failing test that simulates two Claude sessions targeting the same `pluginData` and confirms one of (a) broker reuse (both sessions talk to the same broker), (b) proper lock-based serialization (second session waits). Then implement.

**Risk:** race conditions. Test harness needs to simulate concurrent spawns carefully.

### 2. Structured-output enforcement beyond prompt contract

**Current state:** `parseStructuredOutput` in `lib/copilot.mjs` is best-effort. If the review JSON is malformed, `renderReviewResult` renders a fallback section with parse errors + raw output. The schema at `plugins/copilot/schemas/review-output.schema.json` is only enforced by prompt contract ("return JSON matching this schema").

**Design sketch:** wrap the schema in a validator. Two shapes:
- **A:** validate-and-render — render a "schema violation" report with the validation errors.
- **B:** validate-and-retry — feed the validation errors back to Copilot as a retry prompt (requires multi-turn orchestration).

**Key decision:** plugin currently has zero runtime deps. Adding AJV would be the first. Weigh against hand-rolling a narrow validator for the specific schema shape.

**Key files:**
- `plugins/copilot/scripts/lib/copilot.mjs::parseStructuredOutput`
- `plugins/copilot/scripts/lib/render.mjs::renderReviewResult`
- `plugins/copilot/schemas/review-output.schema.json`

### 3. Custom sandboxing per ACP session

**Upstream-blocked.** `copilot --acp` doesn't expose per-session permission flags. `DEFAULT_COPILOT_SPAWN_ARGS` (`acp-client.mjs`) pins `--allow-all-tools --allow-all-paths --allow-all-urls` at broker spawn. Per-session `--write` toggling is currently enforced by prompt contract only.

**Action:** track upstream Copilot CLI releases. When per-session flags appear, revisit.

## Other non-v1.1 seeded follow-ups

- **Windows production `--effort` for review path:** on Windows prod, `shell:true` + `.cmd` launcher means review prompts (containing XML tags) trip the v0.6 deny-list. Proper fix: resolve `.cmd` to an absolute path and use `shell:false` on Windows. Would need `which`-style PATH + PATHEXT lookup. Not blocking — review via broker (no `--effort`) works fine.
- **Setup-time JSON output consumers:** `/copilot:setup --probe-models --json` now emits `report.modelProbe`. If Claude Code ever grows a UI that wants to pre-warn users, this is the seam to plug into.
