## Active conventions worth preserving

### Branch naming
- `v<N>-<short-name>` for feature branches (e.g. `v10-haiku-fallback-tier`).
- `v<N>-release-bump` for the chore release PR.

### Release process
```bash
# After feature PRs merged to main:
node scripts/bump-version.mjs 0.0.N       # bumps 3 version strings (package-lock.json is optional per v0.4 PR #14)
# Edit plugins/copilot/CHANGELOG.md: prepend a new 0.0.N section with Added/Changed/Fixed bullets
git add package.json plugins/copilot/.claude-plugin/plugin.json .claude-plugin/marketplace.json plugins/copilot/CHANGELOG.md
git commit -m "chore(release): bump version to 0.0.N"
git push -u origin v<N>-release-bump
gh pr create ...                          # release PR
# After CI green on release PR:
gh pr merge <n> --squash --delete-branch --admin
git checkout main && git pull
MERGE_SHA=$(git rev-parse HEAD)
git tag -a v0.0.N $MERGE_SHA -m "v0.0.N — <one-line theme>"
git push origin v0.0.N
gh release create v0.0.N --title "v0.0.N" --notes "$(<CHANGELOG excerpt>)"
```

### Test spawn pattern
- **Always** `run(process.execPath, ...)` for Node-in-tests, never bare `"node"`. The Windows shell wrapper in `tests/helpers.mjs` splits on spaces in the absolute path (`OneDrive - Microsoft` → `OneDrive`), causing `MODULE_NOT_FOUND`. See v0.4 PR #14.

### Fake Copilot (`tests/fake-copilot.mjs`) capabilities
- `--version` probe (short-circuit exit 0, skipped by SPAWN_LOG).
- `--acp` mode (full ACP v1 JSON-RPC; driven by `FAKE_COPILOT_SCRIPT`).
- `-p <prompt>` CLI mode (mirrors real `copilot -p`; emits concatenated `agent_message_chunk` text).
- `FAKE_COPILOT_SPAWN_LOG` → writes one JSON line per spawn with argv + selected env.
- `script.unavailableModels: [...]` → in `-p` mode, invoked models in this list exit 1 with `"Error: model X is not available on this Copilot account."` (targets `isModelUnavailableStderr` regex).

### Spawn safety (cross-spawn helper, v0.0.18)
`safeSpawn` in `plugins/copilot/scripts/lib/safe-spawn.mjs`:
- Drop-in `child_process.spawn` replacement; pre-resolves Windows `.cmd`/`.bat` launchers via PATHEXT and applies cross-spawn-style argv escape with `windowsVerbatimArguments: true`. All three spawn sites (`SpawnedCopilotAcpClient.initialize`, `runCopilotCli`, `probeSingleModel`) use `shell: false`.
- Replaces the v0.0.3-era `SHELL_METACHAR_RE` + `assertNoShellMetachars` deny-list (was incomplete — missed `(`, `)`, `*`).
- Pinned `--deny-tool=shell(<cmd>:*)` for `curl`/`wget`/`nc`/`ncat`/`ssh` on broker `DEFAULT_COPILOT_SPAWN_ARGS` and one-shot `runCopilotCli` (denial > `--allow-all-tools` per `copilot help permissions`).
- `engines.node` bumped to `>=18.20.2 || >=20.12.2 || >=22.0.0` so the CVE-2024-27980 patch is guaranteed present.

### `--effort` (native passthrough since v0.0.16; mapping table removed)
- `--effort low|medium|high|xhigh` is forwarded verbatim to Copilot CLI's
  native `--effort` flag (1.0.11+). The plugin's `none`/`minimal` aliases
  collapse to `low` at spawn time for codex-plugin-cc command parity.
- `--model` and `--effort` are independent: pass either or both. Copilot's
  own runtime applies them independently.
- `--effort` alone (no `--model`) preserves the user's default model from
  `~/.copilot/settings.json` — the legacy mapping that used to force a
  Claude model is gone.
- On `--resume-last`, both `--model` and `--effort` are dropped together
  with one combined stderr notice (broker session can't switch either).
- `EFFORT_TO_MODEL`, `EFFORT_FALLBACK_CHAIN`, `applyEffortFallbackModel`,
  `buildEffortModelChain` were removed in v0.0.16. The model-availability
  fallback retry loop is gone — Copilot's own error surfaces directly.
- `isModelUnavailableStderr` is still exported (used by
  `probeSingleModel` for `--probe-models`) but is no longer wired into
  the runtime path.

### Availability detection
`isModelUnavailableStderr(text)` in `lib/copilot.mjs`:
- Requires `\bmodel\b` + a specific availability phrase on the same line within ~80 chars.
- Bare `forbidden` is **not** an indicator (content-policy false positives).
- Unit-tested in `tests/is-model-unavailable-stderr.test.mjs` (positive + negative cases).

### Broker coordination (v0.0.8)
`plugins/copilot/scripts/lib/broker-lifecycle.mjs`:
- `broker.lock` via `fs.openSync(path, "wx", 0o600)` (atomic `O_CREAT|O_EXCL`). Contents: `<pid>\n<timestamp>\n`.
- Acquirers poll at 25 ms; default 5 s budget.
- Stale-lock recovery via `process.kill(pid, 0)` ESRCH check. `EPERM` treated as alive (holder is another user's process).
- Fast path (live `broker.json` + reachable endpoint) skips the lock entirely — preserves zero-overhead single-session reuse.
- **Dual-budget liveness**: `isBrokerEndpointReady(endpoint, timeoutMs)` — fast path 150 ms, slow path 1000 ms before destructive teardown.

### Review-output validation (v0.0.8, tightened in v0.0.9)
`plugins/copilot/scripts/lib/render.mjs`:
- `validateReviewOutput(data)` walks `plugins/copilot/schemas/review-output.schema.json` end-to-end and returns a list of violations.
- Hand-rolled — no AJV dep. Faithful to the one schema.
- Renderer emits a bulleted `Schema violations:` section so every breakage surfaces in one pass (no first-error short-circuit).
- `minLength: 1` string checks on `summary` / `findings[*].title|body|file` / `next_steps[i]` all use `.trim().length === 0` (v0.0.9 alignment — previously asymmetric).

## Upstream-blocked

These items need Copilot CLI changes before we can act. Track upstream releases; revisit when unblocked.

- **Per-ACP-session sandboxing** (was v1.1 item 3). `copilot --acp` still doesn't expose per-session permission flags. The v0.0.9 upstream audit confirmed per-tool / per-URL flags exist (`--deny-tool`, `--excluded-tools`, `--available-tools`, `--deny-url`, `--allow-url`), but they're spawn-time-global on the broker — not per-session. `DEFAULT_COPILOT_SPAWN_ARGS` pins `--allow-all-tools --allow-all-paths --allow-all-urls`. Unblocks when the flags become ACP `session/new`-reachable.

- **Windows production `--effort` on the review path.** Review prompts contain XML tags and quotes; on Windows production `shell:true` is needed for `.cmd` PATHEXT resolution; combining the two trips the v0.0.6 shell-metachar deny-list. The v0.0.9 upstream audit found **no stdin / `--prompt-file` support** for `-p` — it only accepts positional text. Until Copilot CLI adds stdin or a prompt-file flag, the practical fix would be a Windows-specific cmd.exe quoter (multi-PR scope; we'd replicate Node's pre-CVE-2024-27980 behavior manually, which is exactly the hard problem the CVE exists about). Not blocking for users — review via the broker (no `--effort`) works fine on Windows.

### 2026-04-29 deep-dive: `--output-format json` (DEFERRED)

A 4-agent audit before scoping this refactor surfaced enough risk to defer it:
- **Schema is undocumented and explicitly experimental.** The Copilot CLI `--help` says "JSONL, one JSON object per line", but `docs/cli-programmatic-reference` does not document the event types or field shapes. Issue github/copilot-cli#1482 is open against documentation precisely because tool names like `glob`, `grep`, `view` lack a documented allow-pattern syntax.
- **`copilot help permissions` final paragraph:** *"It is expected that these permissions will be extended in the very near future to support wildcard matching and other richer functionality"* — i.e. forward-compat is explicitly disclaimed for the JSONL surface too.
- **Test fixture rewrite is non-trivial.** `tests/fake-copilot.mjs`'s `-p` mode emits text via concatenated `agent_message_chunk` text. Switching to JSONL emission would touch ~150 LOC of fixture plus 8-12 runtime tests that assert on `result.stdout` content (vs. argv).
- **Backward compat:** today's `parseStructuredOutput` is forgiving (strips fences, returns fallback on JSON failure). Strict per-line JSONL parsing loses that tolerance.

**Recommendation:** revisit when github/copilot-cli ships a documented stable schema (or pins a version in `--output-format json`'s `--help`). Until then, the current text-scraping is "good enough" — `parseStructuredOutput` only runs on the structured-review path, and the schema validator at `validateReviewOutput` already absorbs malformed output cleanly.

### 2026-04-29 deep-dive: granular `--allow-tool` allowlist (PARTIALLY SHIPPED)

The same audit found one CRITICAL blocker for full granular allowlisting:
- `--allow-tool` is **spawn-time-global on the broker**, not per-session. Using different allowlists for review vs. rescue paths would force either (a) two brokers per Claude session (kills the v0.0.8 broker-reuse design), (b) lowest-common-denominator allowlist (defeats the purpose), or (c) inline spawn per invocation (kills performance).

**Shipped in v0.0.18 (cross-spawn refactor):** A follow-up deep-research pass surfaced that the original `--deny-tool` failure was actually shell:true on Windows + cmd.exe paren parsing, NOT a Copilot CLI issue. v0.0.18 introduces `lib/safe-spawn.mjs` (cross-spawn-style helper that pre-resolves Windows `.cmd`/`.bat` launchers and applies cross-spawn's escape pipeline, so cmd metachars in argv reach the child verbatim). With safeSpawn in place, `--deny-tool=shell(curl:*)` etc. work on Windows production: pinned for `curl`/`wget`/`nc`/`ncat`/`ssh` on both broker `DEFAULT_COPILOT_SPAWN_ARGS` and the one-shot CLI's `runCopilotCli` argv. Side effects: `assertNoShellMetachars`/`SHELL_METACHAR_RE` deleted (incomplete deny-list, missing `(`, `)`, `*`); `tests/shell-metachar-regex.test.mjs` deleted (-72 LOC); engines bumped to `>=18.20.2 || >=20.12.2 || >=22.0.0` to guarantee the CVE-2024-27980 patch; new `tests/safe-spawn.test.mjs` finally puts the production Windows `.cmd`-launcher path under CI coverage.

**Still parked (not shipped):** Replacing `--allow-all-tools` with a positive-allowlist of `shell(git:*)`, `shell(gh:*)`, `write`, `view`, `glob`, `grep`. Blocked on per-session allowlist support upstream (see CRITICAL note above). When `session/new` exposes per-session permission flags, this becomes feasible.
- **New env vars (BYOK + ops):** `COPILOT_OFFLINE`, `COPILOT_PROVIDER_BASE_URL` / `_TYPE` / `_API_KEY` / `_BEARER_TOKEN` / `_WIRE_API` / `_AZURE_API_VERSION` / `_MODEL_ID` / `_WIRE_MODEL` / `_MAX_PROMPT_TOKENS` / `_MAX_OUTPUT_TOKENS`, `COPILOT_CUSTOM_INSTRUCTIONS_DIRS`, `COPILOT_AUTO_UPDATE`, `COPILOT_GH_HOST`, `COPILOT_OTEL_*`. ### Narrow DEFAULT_COPILOT_SPAWN_ARGS via granular `--allow-tool` patterns

Surfaced by the 2026-04-21 audit. Copilot CLI 1.0 ships an expanded tool-pattern syntax (`--allow-tool='shell(git:*)'`, `--deny-tool='MyMCP(denied_tool)'`) that would let us replace `--allow-all-tools` in `plugins/copilot/scripts/lib/acp-client.mjs::DEFAULT_COPILOT_SPAWN_ARGS` with an explicit allowlist. This is still spawn-time-global (not per-session — v1.1 item 3 stays blocked), but would tighten the blast radius for prompt-injection scenarios described in `SECURITY.md`'s "What the plugin does *not* defend against".

Design decisions needed before a PR:
- Which tools should be on the allowlist? `shell` for running tests, `write` for making edits, `MyMCP(*)` for MCP servers — but the specific set depends on what `/copilot:task` users expect to just work.
- How does the review path interact? `review` / `adversarial-review` don't need `write`, so a narrower allowlist for review-spawned brokers would be an even bigger win.
- Would a tighter default break any existing user flow? Needs either a toggle or a pre-announce.

Not urgent; parked pending user signal that the current `--allow-all-*` defaults are a problem in practice.
